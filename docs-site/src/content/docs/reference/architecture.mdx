## Workspace de crates

ingenierIA TUI es un workspace Rust con 6 crates:

| Crate | LOC | Responsabilidad |
|-------|-----|-----------------|
| `ingenieria-tui` (binario) | ~37K | Aplicacion principal: app handlers, UI, workers, services |
| `ingenieria-domain` | ~1.8K | Tipos del dominio. Zero dependencias internas |
| `ingenieria-ui` | ~1.7K | 6 themes (TokyoNight default, Solarized, Gruvbox, Monokai, Matrix, HighContrast), design system, primitivas, a11y, frame throttle |
| `ingenieria-api` | ~1.3K | ChatEvent, ToolDefinition, ModelInfo, pricing, retry, metrics |
| `ingenieria-runtime` | ~2.4K | Session, audit, memory, config validation, permisos |
| `ingenieria-tools` | ~1.4K | Bash + 7 validators, hooks types, MCP truncation |

## Patron Action-Reducer

ingenierIA TUI sigue un patron **Elm-like Action-Reducer** con flujo de datos unidireccional:

```
Workers (tokio::spawn) ──> Action ──> MPSC Channel ──> App::handle() ──> AppState ──> render()
```

### Flujo de datos

1. **Workers** corren como tareas async (`tokio::spawn`) y producen valores `Action`
2. **Actions** se envian a traves de un canal MPSC (capacidad: 100)
3. **App::handle()** es el reducer — el unico lugar que muta `AppState`
4. **render()** recibe `&AppState` (inmutable) y dibuja la UI

## Responsabilidades por modulo

| Modulo | Responsabilidad |
|--------|-----------------|
| `app/` | 35 handler files del reducer por pantalla. Unico lugar que muta estado |
| `state/` | `AppState` con 15+ sub-modulos. Solo tipos y defaults |
| `actions.rs` | Enum `Action` (~120+ variantes). Solo tipos |
| `domain/` | Re-exports de `ingenieria-domain`. Sin logica |
| `services/` | 41+ modulos: HTTP, AI providers, tools, MCP, agents, bridge, cron, lsp, plugins, recovery, onboarding, doctor, structured output |
| `ui/` | 18+ modulos de renderizado puro con ratatui. Recibe `&AppState` inmutable |
| `workers/` | 10 tareas async de larga duracion. Solo envian Actions |
| `registries/` | Plugin registry + Agent registry |
| `utils/` | Helpers de texto (truncate, visible_width) |
| `config.rs` | Resolucion de configuracion con Levenshtein validation |

## Separacion `/` slash vs `:` palette

Invariante del dispatcher:

- `/` (42 comandos) → **solo chat**: sesion, contexto AI, agents, todos, memoria, workflows.
- `:` (26 comandos) → **config/ops/nav**: sync, diagnostico, navegacion, exploradores, configuracion, instalacion.

Si un slash eliminado se escribe, el reducer responde con hint de redireccion (ver `is_migrated_to_palette`).

## Reglas criticas

- `ui/` **NUNCA** importa de `services/`
- `domain/` no depende de nada interno
- Maximo **400 lineas** por archivo — dividir por responsabilidad si se excede
- Maximo **40 lineas** por funcion — extraer subfunciones
- No `.unwrap()` en codigo de produccion — usar `?` o `unwrap_or`
- Sin efectos secundarios en funciones de render (`ui/*.rs`)
- Sin allocaciones en heap en render salvo que sea inevitable — preferir `&str`, `Cow<str>`
- Todo estado observable vive en `AppState` — sin estado externo
